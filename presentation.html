❗ Problem Statement

Why Does This System Exist?

📋
Information Silos
Ward nurses log patients on paper, doctors record diagnoses on prescription pads, and billing departments compile invoices from scratch — causing transcription errors and data loss.
📅
Scheduling Conflicts
Appointments are booked without visibility into doctor availability, leading to overbooking, long queues, and clinician fatigue across departments.
💸
Billing Omissions
Completed consultations must be manually reported to the billing counter. Patients leaving without notification results in direct revenue loss.
📉
No Live Metrics
Clinic managers have zero real-time insight into daily doctor load, patient demographics, diagnostic patterns, or revenue indicators.
🎯 Objectives

Project Objectives

Integrated Operations
Bridge gaps between patient registration, doctor scheduling, appointment logging, and invoicing in a single synchronized platform.
Automated Invoicing
Business rule engine that auto-triggers a ₹150 billing record whenever a doctor marks a consultation as COMPLETED.
Persistent Data Storage
Normalized database schema via Spring Data JPA & H2 file-based storage — zero database setup, data preserved across restarts.
Premium UI Dashboard
Modern SPA with real-time KPI metrics, age & revenue distribution charts, and glassmorphic dark/light theming.
Clerical Efficiency
Enable ward managers, billing clerks, and medical officers to quickly search, schedule, update, and audit clinical records.

🏗️ System Architecture

Three-Layer Architecture

🌐 SPA Frontend (HTML + CSS + JS)
↕ REST / JSON (AJAX fetch)
⚙️ Spring Boot REST Controllers
↕ @Autowired Repositories
🗃️ Spring Data JPA + Hibernate
↕ JDBC SQL
💾 H2 File Database (hospital_db)
Controller Layer
@RestController classes handle HTTP routing, request validation and response serialization for all 5 modules.
Repository Layer
JpaRepository interfaces auto-generate CRUD SQL — no boilerplate needed. Custom queries via method naming.
Model Layer
@Entity annotated POJOs define database tables. @ManyToOne relationships link Appointments to Patients & Doctors.

👤 Module 1

Patient Management

📋 Features
  • Register new patients with full demographic and medical history
  • View, search, and filter patient records in real-time
  • Update patient details (address, contact, medical notes)
  • Delete patient records with cascade handling
  • Seeded with 5 sample patients on first startup
✅ Validation Rules
  • Name: @NotBlank (required)
  • Email: @Email (valid format enforced)
  • Phone: @NotBlank (required)
  • Gender: @NotBlank (required)
SAMPLE PATIENT ENTITY
@Entity class Patient {
  @Id @GeneratedValue
  private Long id;
  @NotBlank
  private String name;
  @Email
  private String email;
  private String gender;
  private String phone;
  private String medicalHistory;
}

🌱 Data Seeder & Testing

Startup Data Seeder & Validation

🌱 DataSeeder Component
  • Implements CommandLineRunner — runs on every app startup
  • Checks if doctorRepository.count() == 0 before seeding
  • Seeds 5 Doctors, 5 Patients, 5 Appointments & 4 Billing records
  • Data reflects real clinical scenarios (diagnoses, prescriptions)
  • Idempotent — will not duplicate data on restarts
🧪 Testing Strategy
  • Spring Boot context load test via @SpringBootTest
  • JUnit 5 for unit and integration test coverage
  • Maven Surefire plugin runs tests on every build
  • Real-world evaluation at 2 hospitals in Raipur
  • Build verified: mvnw clean package — BUILD SUCCESS ✅
